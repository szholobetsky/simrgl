# SIMARGL: Метрики структурної цілісності для адаптивного навчання на графах залежностей

## Огляд проєкту

**SIMARGL** – це дослідницький фреймворк для оцінювання та балансування рекомендацій щодо змін коду. Система рекомендує зв'язки між модулями, файлами або функціями, які слід змінити разом під час впровадження нових вимог.

Основна інновація – це **пар метрик, що суперечать одна одній**, які вимірюють компроміс між:
- **Новизною** – рекомендаціям НОВИХ зв'язків (інновація)
- **Структурністю** – збереження ІСНУЮЧОЇ модульної структури (стабільність)

### Етимологія

Назва отримана від Simargl (Симаргл) – слов'янського крилатого бога у формі собаки, який охороняв Дерево Життя та доставляв повідомлення між богами та смертними. Його довгий хвіст іноді заплутувався, що призводило до неправильних повідомлень – метафора про компроміс між доставкою нової інформації (новизна) та підтримкою структурного порядку (цілісність).

---

## Визначення проблеми

Коли система рекомендацій пропонує зміни коду, вона може робити два типи помилок:

1. **Занадто консервативна** – рекомендує лише існуючі, добре відомі зв'язки
   - Висока точність, але немає цінності (проблема "розбитого годинника")
   - Система стагнирует, немає інновацій
2. **Занадто нав'язлива** – рекомендує багато нових зв'язків між модулями
   - Висока новизна, але руйнує модульну структуру
   - Створює архітектуру "spaghetti code"

**Мета**: Знайти рекомендації, які є ОДНОЧАСНО новими ТА поважають існуючу модульну структуру.

---

## Основні метрики

### Novelty@K

**Визначення**: Фракція рекомендованих зв'язків, які є НОВИМИ (не входять до існуючого коду).

```
Novelty@K = |{r ∈ top-K : r ∉ ExistingRelations}| / K
```

- **Високе значення (→1.0)**: Система рекомендує переважно нові зв'язки
- **Низьке значення (→0.0)**: Система рекомендує переважно існуючі зв'язки

**Інтерпретація**:
- Novelty@K = 1.0 означає, що всі рекомендації нові (можливо, ризиковано)
- Novelty@K = 0.0 означає, що всі рекомендації вже існують (не додають цінності)

---

### Structurality@K

**Визначення**: Фракція рекомендованих зв'язків, де ОБОД оборот та кінцева точка належать ОДНОМУ модулю.

```
Structurality@K = |{r ∈ top-K : module(r.source) == module(r.target)}| / K
```

- **Високе значення (→1.0)**: Рекомендації залишаються в межах модуля
- **Низьке значення (→0.0)**: Рекомендації перетинають межі модулів

**Інтерпретація**:
- Structurality@K = 1.0 означає, що всі рекомендації є міжмодульними (зберігають структуру)
- Structurality@K = 0.0 означає, що всі рекомендації є міжмодульними (можуть порушити структуру)

## 2×2 матриця компромісів

Об'єднання Novelty та Structurality створює чотири зони:

```
                          Novelty@K
                     (нові зв'язки)
                          ВИСОКА (1.0)
                             ▲
                             │
        ┌────────────────────┼────────────────────┐
        │                    │                    │
        │     ПОРОШЕННЯ     │     ЕВОЛЮЦІЯ      │
        │                    │                    │
        │   Нові + міжмодульні   │   Нові + в межах   │
        │   посилання     │   модулів     │
        │                    │                    │
        │   ⚠️ НЕОБХІДНО     │   ✅ ІДЕАЛЬНО      │
        │                    │                    │
Низька ◄───┼────────────────────┼────────────────────┼───► Висока
(0.0)   │                    │                    │    (1.0)
        │    СТАГНІЯ    │    ДОБРОДІЙ    │  Structurality@K
        │                    │                    │  (в межах модулів)
        │   Існуючі + міжмодульні   │   Існуючі + в межах   │
        │   посилання     │   модулів     │
        │                    │                    │
        │   ❌ НЕКОШТОВНО    │   🔧 БЕЗПЕЧНО     │
        │                    │                    │
        └────────────────────┼────────────────────┘
                             │
                             ▼
                          Низька (0.0)
```

### Опис зон

| Зона | Novelty | Structurality | Опис | Дія |
|------|---------|---------------|-------------|--------|
| **ЕВОЛЮЦІЯ** | Висока | Висока | Нові зв'язки, що поважають межі модуля | ✅ Ідеальний стан – слід дотримуватися |
| **ПОРОШЕННЯ** | Висока | Низька | Нові зв'язки, що перетинають межі модуля | ⚠️ Може порушити архітектуру |
| **ДОБРОДІЙ** | Низька | Висока | Існуючі зв'язки в межах модуля | 🔧 Безпечно, але немає інновацій |
| **СТАГНІЯ** | Низька | Низька | Існуючі міжмодульні зв'язки | ❌ Без цінності, слід уникати |

## Виведені метрики

### Метрики для конкретних зон

Кожна рекомендація належить точно до однієї зони:

```python
Evolution@K    = |{r ∈ top-K : r є НОВОЮ і r є МІЖМОДУЛЬНОЮ}| / K
Disruption@K   = |{r ∈ top-K : r є НОВОЮ і r є МІЖМОДУЛЬНОЮ}| / K
Maintenance@K  = |{r ∈ top-K : r є ІСНУЮЧОЮ і r є МІЖМОДУЛЬНОЮ}| / K
Stagnation@K   = |{r ∈ top-K : r є ІСНУЮЧОЮ і r є МІЖМОДУЛЬНОЮ}| / K
```

**Обмеження**: Evolution@K + Disruption@K + Maintenance@K + Stagnation@K = 1.0

### Комбіновані метрики

**Показник структурної еволюції (SES)**:
```python
SES = sqrt(Novelty@K × Structurality@K)
```
- Діапазон: [0, 1]
- Штрафує, якщо будь-яка з метрик низька
- Максимальний, коли обидві високі

**Гармонійний показник еволюції (HES)**:
```python
HES = 2 × Novelty@K × Structurality@K / (Novelty@K + Structurality@K)
```
- Більш чутливий до дисбалансу, ніж SES

## Чому важливий баланс

### Проблема упередження популярності

В реальних кодових базах, невеликі "основні" модулі часто змінюються дуже часто:

```
Розподіл змін модуля:
  CoreUtils (45%) ████████████████████████████████████
  DataLayer (25%) ████████████████████
  UI (15%) ████████████
  Auth (10%) ████████
  Config (5%) ████
```

Наївна система завжди рекомендує CoreUtils, тому що:
- Воно з'являється у 45% усіх змін
- Рекомендація CoreUtils дає 45% "точності"
- Але це проблема "розбитого годинника" – іноді правильно, але марно

### Проблема "spaghetti code"

Система, оптимізована лише для новизни, може рекомендувати:

```
Завдання: "Додати автентифікацію користувача"
Рекомендації:
  1. PaymentService → AuthController (НОВА, міжмодульна) ✅
  2. ReportGenerator → UserModel (НОВА, міжмодульна) ✅
  3. CacheManager → SessionService (НОВА, міжмодульна) ✅
```

Всі нові зв'язки! Висока новизна! Але це створює тісний зв'язок між непов'язаними модулями.

### Ідеальний результат

Балансована система рекомендує:

```
Завдання: "Додати автентифікацію користувача"
Рекомендації:
  1. AuthController → UserValidator (НОВА, в межах модуля auth) ✅
  2. AuthController → TokenService (НОВА, в межах модуля auth) ✅
  3. UserService → AuthController (НОВА, міжмодульна, але логічна) ⚠️
```

- Нові зв'язки (новизна)
- Переважно в межах модуля auth (структурність)
- Зв'язок між модулями, лише коли це логічно

## Рекомендації щодо реалізації

### Обчислення метрик

```python
class SIMARGLMetrics:
    def __init__(self, existing_relations: Set[Tuple], module_map: Dict[str, str]):
        """
        existing_relations: Set of (source, target, relation_type) tuples
        module_map: Dict mapping function/file ID to module ID
        """
        self.existing = existing_relations
        self.modules = module_map
    
    def is_novel(self, relation) -> bool:
        return (relation.source, relation.target, relation.type) not in self.existing
    
    def is_intra_module(self, relation) -> bool:
        src_module = self.modules.get(relation.source)
        tgt_module = self.modules.get(relation.target)
        return src_module is not None and src_module == tgt_module
    
    def compute_metrics(self, recommendations: List, k: int) -> Dict:
        top_k = recommendations[:k]
        
        novel = [r for r in top_k if self.is_novel(r)]
        intra = [r for r in top_k if self.is_intra_module(r)]
        
        novelty = len(novel) / k
        structurality = len(intra) / k
        
        evolution = len([r for r in novel if self.is_intra_module(r)]) / k
        disruption = len([r for r in novel if not self.is_intra_module(r)]) / k
        maintenance = len([r for r in top_k if not self.is_novel(r) and self.is_intra_module(r)]) / k
        stagnation = len([r for r in top_k if not self.is_novel(r) and not self.is_intra_module(r)]) / k
        
        ses = (novelty * structurality) ** 0.5
        
        return {
            'Novelty@K': novelty,
            'Structurality@K': structurality,
            'Evolution@K': evolution,
            'Disruption@K': disruption,
            'Maintenance@K': maintenance,
            'Stagnation@K': stagnation,
            'SES': ses
        }
```

### Використання температури для балансування

Використовуйте параметр температури для контролю балансу між новизною та структурністю:

```python
def balanced_score(semantic_score, novelty_score, structural_score, temperature=1.0):
    """
    temperature < 1.0: Favors structure (conservative)
    temperature = 1.0: Balanced
    temperature > 1.0: Favors novelty (explorative)
    """
    novelty_weight = temperature / (1 + temperature)
    structure_weight = 1 / (1 + temperature)
    
    combined = (
        0.5 * semantic_score +
        novelty_weight * novelty_score +
        structure_weight * structural_score
    )
    return combined
```

## Науковий внесок

### Що нового в SIMARGL

1. **Метрики структурної цілісності (Structurality@K)** – вимірюють вплив рекомендацій на структуру модуля
2. **Антагоністична пара метрик** – новизна проти структурності
3. **2×2 декомпозиція** – зони ЕВОЛЮЦІЯ/ПОРОШЕННЯ/ДОБРОДІЙ/СТАГНІЯ
4. **Застосування до рекомендацій щодо змін коду** – перенесення з теор পানিতে програмування
5. **Визначення модуля** – одиниця коду (пакет, namespace, директорія)
6. **Міжмодульний** – зв'язок, що перетинає межі модуля
7. **Новий** – зв'язок, що не існує в поточному коді
8. **Зворотний** – зв'язок, що залишається в межах модуля
9. **Еволюція** – новий + міжмодульний (ідеальний)
10. **Порушення** – новий + міжмодульний (небезпечний)
11. **Добродій** – існуючий + міжмодульний (безпечний)
12. **Стагнація** – існуючий + міжмодульний (небезпечний)
