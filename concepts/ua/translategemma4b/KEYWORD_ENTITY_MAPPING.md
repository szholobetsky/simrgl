## Ключова ідентифікація для покращеної RAG контексту

## Основна проблема

### Поточні обмеження: Роз'єднаний контекст

Поточна реалізація RAG страждає від **поганої передбачуваності**, тому що:

1. **RECENT vs ALL пошуки повертають непересічні результати** - Різні колекції створюють повністю різні набори файлів/модулів/завдань без перекриття.
2. **Відсутнє структурне розуміння** - LLM отримує "переплетений" список потенційно релевантних файлів, не розуміючи їх зв'язків.
3. **Відсутня карта проєкту** - Система не має високорівневого розуміння, де знаходяться сутності та концепції у кодовій базі.
4. **Слабкий нагляд** - Невеликий локальний LLM не може вирішувати складні зв'язки з неструктурованими, роз'єднаними даними.

### Аналогія

> **Поточний підхід**: "Створіть шлях від Пари до Лондона, а ось тут є випадкові назви сіл, що можуть бути десь між ними"
>
> **Необхідний підхід**: "Ось карта з Пари, Лондоном, великими містами (ключовими словами/сутностями) та дорогами, що їх з'єднують"

## Рішення: Ключово-орієнтоване відображення сутностей

### Ключовий момент від людського розуміння

Коли читаєте опис завдання, наприклад:

```
"Не можна шукати слова довші за 15 символів в описі правил"
```

Людина одразу розпізнає **"RULE"** як ключове слово, тому що:

1. **RULE - це сутність** - Основний доменний концепт/елемент зберігання в системі.
2. **RULE зустрічається в багатьох місцях** - Не є одноразовою появою.
3. **RULE вбудований у структуру коду** - Відображається в шляхах файлів, назвах класів, назвах методів.

Системі потрібно автоматично ідентифікувати та відображати ці **бізнес-терміни** на **місця в коді**.

## Академічна база

Ця ідея добре підкріплена великою кількістю досліджень в області розробки програмного забезпечення:

### 1. Вилучення доменних концепцій із коду

**Основний висновок**: Ідентифікатори програм містять доменні концепції, які можна вилучити за допомогою NLP.

> "Розуміння програми полягає у відображенні доменних концепцій на елементи коду, які їх реалізують. Назви ідентифікаторів містять відповідні підказки для відновлення відображення та надання його програмістам."

**Дослідження**:
- [Extraction of domain concepts from the source code (ScienceDirect, 2014)](https://www.sciencedirect.com/science/article/pii/S0167642314004419)
- [Towards the Extraction of Domain Concepts from the Identifiers (IEEE, 2011)](https://ieeexplore.ieee.org/document/6079777/)
- [On the Use of Domain Terms in Source Code (ResearchGate, 2008)](https://www.researchgate.net/publication/4349667_On_the_Use_of_Domain_Terms_in_Source_Code)

### 2. Локалізація та відображення функцій

**Основний висновок**: Методи пошуку інформації можуть відображати природні мовні концепції на місця в коді.

> "Локалізація концептів ідентифікує частини програмної системи, які реалізують певну концепцію, що походить від проблеми або домену рішення."

**Дослідження**:
- [An information retrieval approach to concept location in source code (IEEE, 2003)](https://ieeexplore.ieee.org/document/1374321/)
- [Feature location in source code: A taxonomy and survey (Wiley, 2013)](https://onlinelibrary.wiley.com/doi/full/10.1002/smr.567)
- [Text Retrieval Approaches for Concept Location in Source Code (Springer, 2013)](https://link.springer.com/chapter/10.1007/978-3-642-36054-1_5)
- [Concept location using formal concept analysis and information retrieval (ACM TOSEM, 2012)](https://dl.acm.org/doi/10.1145/2377656.2377660)

### 3. Відстеження вимог

**Основний висновок**: Семантичний аналіз може заповнити прогали між бізнес-вимогами та сутностями в коді.

> "Традиційні методи відстеження, які покладаються на текстові подібності між вимогами та кодом, часто страждають від низької точності через семантний простір між вимогами на високому рівні та синтаксичну природу коду."

**Дослідження**:
- [An empirical study on the importance of source code entities for requirements traceability (Springer, 2015)](https://link.springer.com/article/10.1007/s10664-014-9315-y)
- [Recovering semantic traceability between requirements and design (Springer, 2019)](https://link.springer.com/article/10.1007/s11334-019-00330-w)
- [On the role of semantics in automated requirements tracing (Springer, 2014)](https://link.springer.com/article/10.1007/s00766-013-0199-y)
- [Constructing Traceability Links between Software Requirements and Source Code Based on Neural Networks (MDPI, 2023)](https://www.mdpi.com/2227-7390/11/2/315)

### 4. Проблема словника

**Основний висновок**: Найбільший виклик полягає в тому, що розробники та вимоги використовують різні терміни для одних і тих самих концепцій.

> "The Vocabulary Problem in Human-System Communication is recognized as a fundamental challenge in this area."

## Підхід до реалізації

### Фаза 1: Вилу összef концептів із коду

```python
async def phase1_reasoning_with_entity_map(task_description):
    # Вилучення ключових слів із опису завдання
    keywords = extract_keywords_nlp(task_description)

    # Пошук сутностей
    entity_boosts = {}
    for keyword in keywords:
        entity = lookup_entity(keyword)
        if entity:
            for file in entity['files']:
                entity_boosts[file] = entity_boosts.get(file, 0) + 0.3

    # Запуск пошуку
    module_results = vector_search_modules(task_description, top_k=10)
    file_results = vector_search_files(task_description, top_k=15)

    # Застосування підвищення до оцінок
    for file in file_results:
        if file['path'] in entity_boosts:
            file['score'] += entity_boosts[file['path']]

    # Пересортування
    file_results = sorted(file_results, key=lambda x: x['score'], reverse=True)

    # Додавання контексту сутностей до промпта LLM
    entity_context = build_entity_context(keywords, entity_map)

    return {
        'files': file_results,
        'entity_context': entity_context
    }
```

## Переваги цього підходу

### 1. Структурний контекст ("Карта")

Замість випадкових сіл, LLM отримує:
- **Сутності**: Основні доменні концепції (RULE, QUALITY_PROFILE, INDEX)
- **Зв'язки**: Як сутності пов'язані (RULE пов'язане з INDEX)
- **Географія**: Де знаходиться кожна сутність у кодовій базі.

### 2. Покращена точність

- **Підвищення ключових слів**: Файли, що відповідають доменним сутностям, отримують вищий пріоритет.
- **Дискредитація**: Коли "rule" з'являється в завданні, система знає, що це посилання на сутність RULE.
- **Фільтрація**: Можна фільтрувати неправдоподібні файли, які мають високу оцінку семантичної схожості, але неправильну сутність.

### 3. Краще розуміння LLM

З "картою" навіть невеликий LLM може:
- Розуміти "RULE - це основна концепція, тому RuleIndex.java є дуже релевантним".
- Розумно аналізувати "Завдання згадує 'опис правил', тому я повинен отримати сутності RULE та DESCRIPTION".
- Виводити "Якщо RULE пов'язане з INDEX, перевірте конфігурацію ElasticSearch".

### 4. Об'єкція до змін коду

- **Рефакторинг**: Якщо RuleIndex.java перейменова на RuleFinder.java, карта автоматично оновлюється.
- **Заостряння**: Можна відновити бізнес-терміни з історичних завдань.
- **Нові функції**: Карта росте, коли з'являються нові доменні концепції.

### 5. Перенесення між проєктами

Коли ви створили конвеєр вилучення сутностей:
- Застосовуйте до будь-якої кодової бази Java/Python/C#
- Це працює для старих кодових баз з гарною ідентифікацією
- Це працює для замаскованої коду, використовуючи історію завдань.

## Висновок

Підхід з вилученням сутностей надає "карту", якої не було в RAG. Вилучаючи доменні сутності з коду, ми можемо:

- Дати LLM **структурний контекст**, а не випадкові файли.
- **Підвищувати пріоритет** релевантних файлів на основі сутностей.
- **Обробляти замаскований код** відновлюючи бізнес-терміни.
- **Покращувати точність** завдяки пошуку, керованому ключовими словами.
- **Дозволити краще розуміння** навіть з невеликими LLM.

Цей підхід **добре підкріплений академічними дослідженнями** і вирішує основну проблему: надання нагляду та структури, щоб LLM міг орієнтуватися в кодовій базі, як програміст із предметними знаннями.
