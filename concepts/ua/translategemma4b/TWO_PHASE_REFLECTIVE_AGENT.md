# Двухфазная архитектура агента с отражением
## Обзор

Этот документ описывает улучшенную архитектуру системы агента RAG-MCP, которая включает в себя возможности саморефлексии и рассуждения для удовлетворения требований финального проекта.

## Общая информация и обоснование

### Ограничения текущей системы

Существующий агент RAG-MCP следует простому пайплайну:
```
Запрос пользователя → Поиск с помощью MCP → Создание контекста → LLM → Вывод
```

**Что он делает хорошо:**
- ✅ Семантический поиск с использованием инструментов MCP
- ✅ Рекомендации кода на основе LLM
- ✅ Интеграция с базой данных векторов PostgreSQL

**Что отсутствует:**
- ❌ Отсутствует явный процесс рассуждения/мышления
- ❌ Нет рефлексии над собственными решениями
- ❌ Нет самопроверки собственного кода
- ❌ Ограниченная прозрачность в процессе принятия решений

### Требования проекта

На основании обратной связи от проекта, агент должен демонстрировать:
1. **Рассуждение** - Явное представление мыслительного процесса и планирования
2. **Действия на основе инструментов** - Выполнение значимых операций (мы это уже делаем с помощью MCP)
3. **Рефлексия** - Критика собственной производительности и решений
4. **Самопроверка** - Анализ собственного исходного кода и предложения по улучшению

## Предлагаемая архитектура: Двухфазный рефлексивный агент

Агент работает в двух отдельных фазах, которые подчеркивают различные когнитивные способности:

### Фаза 1: Выполнение (Рассуждение → Действие → Анализ)

Эта фаза обрабатывает основную задачу с использованием явного рассуждения.

```
Запрос пользователя
    ↓
[1.1 РАССУЖДЕНИЕ]
    - Понять намерения пользователя
    - Разработать подход и использование инструментов
    - Выявить потенциальные проблемы
    - Определить критерии успеха
    ↓
[1.2 ДЕЙСТВИЕ]
    - Выполнять запланированные вызовы инструментов MCP
    - Логировать каждое действие с обоснованием
    - Собирать результаты поиска
    - Собирать соответствующий контекст
    ↓
[1.3 АНАЛИЗ]
    - Оценить качество результатов
    - Оценить соответствие запросу
    - Выявить пробелы в информации
    - Синтезировать выводы
```

### Фаза 2: Мета-рефлексия (Рефлексия → Самопроверка)

Эта фаза добавляет самосознание и способность к непрерывному улучшению.

```
[2.1 РЕФЛЕКСИЯ]
    - Критически оценить фазу выполнения
    - Оценить качество принятия решений
    - Оценить уровень уверенности
    - Выявить, что можно улучшить
    - Решить, нужна ли повторная попытка
    ↓
[2.2 САМОПРОВЕРКА] (по запросу)
    - Читать свой исходный код
    - Анализировать паттерны реализации
    - Выявлять потенциальные ошибки
    - Предлагать архитектурные улучшения
    - Рекомендовать оптимизации
    ↓
Финальный вывод (с полной прозрачностью)
```

## Подробное описание фаз

### Фаза 1.1: Рассуждение

**Цель:** Сделать мыслительный процесс агента явным и осознанным.

**Реализация:**
```python
async def reasoning_phase(self, query: str) -> ReasoningOutput:
    """
    Агент продумывает проблему, прежде чем действовать.
    Возвращает структурированный вывод рассуждения.
    """
    reasoning_prompt = f"""
    Исходный запрос пользователя: "{query}"

    Подумайте шаг за шагом:
    1. Чего на самом деле хочет пользователь?
    2. Какая информация мне нужна?
    3. Какие инструменты MCP мне следует использовать и в каком порядке?
    4. Какие потенциальные проблемы или крайние случаи?
    5. Как я буду оценивать, насколько хорош мой ответ?
    6. Что будет считаться успехом?

    Предоставьте структурированное рассуждение с четким обоснованием решений.
    """

    reasoning = await self.llm.generate(reasoning_prompt)

    return ReasoningOutput(
        user_intent=...,
        planned_actions=[...],
        tool_sequence=[...],
        expected_challenges=[...],
        success_criteria=[...]
    )
```

**Ключевые преимущества:**
- Делает процесс принятия решений прозрачным
- Помогает определить правильные инструменты до выполнения
- Предоставляет трассировку мыслительного процесса агента
- Обеспечивает лучшее управление ошибками

### Фаза 1.2: Действие

**Цель:** Выполнение запланированных действий с подробным отслеживанием.

**Реализация:**
```python
async def action_phase(self, reasoning: ReasoningOutput) -> ActionOutput:
    """
    Выполнение запланированных действий с подробным отслеживанием.
    Каждое действие записывается с его назначением и результатом.
    """
    actions_taken = []

    # Действие 1: Поиск соответствующих модулей
    if "search_modules" in reasoning.tool_sequence:
        modules = await self.mcp.search_modules(
            query=reasoning.user_intent,
            top_k=5
        )
        actions_taken.append(Action(
            name="search_modules",
            parameters={"top_k": 5},
            rationale="Поиск высокоуровневой организации кода",
            result_count=len(modules),
            quality_score=self._assess_results(modules)
        ))

    # Действие 2: Поиск конкретных файлов
    if "search_files" in reasoning.tool_sequence:
        files = await self.mcp.search_files(
            query=reasoning.user_intent,
            top_k=10
        )
        actions_taken.append(Action(
            name="search_files",
            parameters={"top_k": 10},
            rationale="Поиск конкретных файлов реализации",
            result_count=len(files),
            quality_score=self._assess_results(files)
        ))

    # Действие 3: Поиск исторических задач
    if "search_similar_tasks" in reasoning.tool_sequence:
        tasks = await self.mcp.search_similar_tasks(
            query=reasoning.user_intent,
            top_k=5
        )
        actions_taken.append(Action(
            name="search_similar_tasks",
            parameters={"top_k": 5},
            rationale="Обучение на основе прошлых решений",
            result_count=len(tasks),
            quality_score=self._assess_results(tasks)
        ))

    return ActionOutput(
        actions=actions_taken,
        total_results=...,
        execution_time=...,
        encountered_errors=[...]
    )
```

**Ключевые преимущества:**
- Явная отслеживаемость всех операций
- Каждое действие имеет документированное назначение
- Оценка качества для каждого действия
- Лег buildFor debugging
- Оптимизированная обработка ошибок

## Формат ответа

Агент возвращает комплексный, прозрачный ответ:

```python
class AgentResponse:
    """Полный ответ агента с полной прозрачностью"""

    # Исходный запрос пользователя
    query: str

    # Выходы фаз 1
    reasoning: ReasoningOutput
    actions: ActionOutput
    analysis: AnalysisOutput

    # Выходы фаз 2
    reflection: ReflectionOutput
    self_inspection: Optional[SelfInspectionOutput]

    # Синтетический окончательный ответ
    answer: str

    # Метаданные
    confidence_score: float  # 0-100
    processing_time: float
    total_tokens_used: int

    def format_for_display(self) -> str:
        """Форматирование ответа для отображения пользователю"""
        return f"""
        {'='*80}
        ЗАПРОС: {self.query}
        {'='*80}

        РАССУЖДЕНИЕ:
        {self.reasoning.format()}

        Действия, выполненные:
        {self.actions.format()}

        АНАЛИЗ:
        {self.analysis.format()}

        РЕФЛЕКСИЯ:
        {self.reflection.format()}

        {self.self_inspection.format() if self.self_inspection else ''}

        {'='*80}
        ФИНАЛЬНЫЙ ВЫВОД:
        {self.answer}

        Уверенность: {self.confidence_score}%
        Время обработки: {self.processing_time:.2f}с
        {'='*80}
        """
```

## Влияние на модели Ollama с встроенным рассуждением

### Вопрос: Что, если Ollama модели поддерживают рассуждение из коробки?

**Если** будущие Ollama модели включают встроенные "мышление" или "рассуждение" (подобно OpenAI o1), архитектура адаптируется, но не меняется фундаментально:

**Преимущества:**

- Фаза 1.1 (Рассуждение) становится более естественной и эффективной
- Лучшее внутреннее рассуждение
- Потенциально более быстрое выполнение
- Более последовательные цепочки мыслей
- Оптимизация использования токенов

**Изменения:**

```python
async def reasoning_phase(self, query: str) -> ReasoningOutput:
    """
    С встроенным рассуждением: Дать модели думать внутри,
    а затем извлечь структурированное рассуждение.
    """
    # Модель думает внутренне (токены для внутреннего рассуждения)
    response = await self.llm.generate(
        prompt=query,
        enable_thinking=True,  # Новый параметр
        thinking_budget=1000   # Токены для внутреннего рассуждения
    )

    # Извлечь рассуждение из мыслей модели
    reasoning = ReasoningOutput(
        user_intent=response.thinking.user_intent,
        planned_actions=response.thinking.planned_actions,
        # ... извлечено из процесса мышления
    )

    return reasoning
```

**Остается прежним:**

- Архитектура двухфаз остается
- Рефлексия и самопроверка остаются необходимыми
- Явное использование инструментов остается неизменным
- Анализ остается неизменным
- Сохраняется прозрачность

**Ключевой момент:** Встроенное мышление улучшит Фазу 1.1 (Рассуждение), но не устранит необходимость в:

- Фазе 1.2 (Действие) - все еще необходимо явное использование инструментов
- Фазе 1.3 (Анализ) - все еще необходимо оценку качества
- Фазе 2.1 (Рефлексия) - все еще необходимо критическое осмысление
- Фазе 2.2 (Самопроверка) - все еще необходимо анализ кода

Архитектура разработана таким образом, чтобы быть **независимой от режима мышления** - она работает как с:

1. Текущим подходом: Явное запрошение рассуждения
2. Будущим подходом: Встроенные токены мышления

## Сравнение: Один агент против нескольких агентов

### Вариант A: Один двухфазный агент (РЕКОМЕНДУЕТСЯ для Capstone)

**Структура:**

```
TwoPhaseReflectiveAgent
    ├── Фаза 1: Выполнение
        ├── Рассуждение
        ├── Действие
        └── Анализ
    └── Фаза 2: Мета-рефлексия
        ├── Рефлексия
        └── Самопроверка
```

**Преимущества:**
- Простая структура
- Легче отлаживать и поддерживать
- Все логика в одном компоненте
- Четкая последовательность фаз
- Достаточная сложность для академической оценки
- Легче объяснить и документировать

**Недостатки:**
- Один класс может стать большим
- Все логика в одном компоненте
- Меньше модульности для будущего расширения
- Лучше всего подходит для проекта Capstone, proof of concept, образовательных целей

### Вариант B: Три специализированных агента

**Структура:**

```
Оркестратор
    ├── Агент рассуждения (планирует и думает)
    ├── Агент действия (выполняет инструменты MCP)
    └── Агент мета-рефлексии (критикует и проверяет)
```

**Преимущества:**
- Истинная многоагентная архитектура
- Разделение ответственности
- Каждый агент отвечает за определенную задачу
- Более впечатляющая архитектура
- Легче расширять отдельные компоненты
- Возможность параллельной обработки

**Недостатки:**
- Более сложная оркестрация
- Передача данных между агентами
- Больше потенциальных точек отказа
- Трудно отлаживать
- Перебор
- Лучше всего подходит для производственных систем, исследований многоагентных систем, будущего масштабирования
