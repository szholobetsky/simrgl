# SIMARGL Research Project Overview

## Executive Summary

**SIMARGL** (Structural Integrity Metrics for Adaptive Relation Graph Learning) – це комплексний науковий проєкт, що розробляє інтелектуальну систему навігації та рекомендацій коду. Цей проєкт вирішує фундаментальну проблему в інженерії програмного забезпечення: як допомогти розробникам знаходити правильні артефакти коду (модулі, файли, функції) для внесення змін при реалізації нових функцій або виправленні помилок.

Основна інновація полягає в поєднанні:
- **Розширених технік вбудовування** для семантичного розуміння коду
- **Двофагових агентів з рефлексією** для інтелектуального обґрунтування
- **Філософського підґрунтя** для з'єднання людської наміру з семантикою коду
- **Архітектури RAG з двома серверами** для балансування історичних патернів із останнім контекстом

---

## Problem Statement

### The Navigation Challenge

Сучасні програмні системи містять мільйони рядків коду, організованих у сотні або тисячі модулів. Коли розробник отримує завдання, наприклад, "Виправити проблему з автентифікацією", він стикається з кількома викликами:

1.  **З чого почати?** Які модулі мають відношення до автентифікації?
2.  **Що пов'язано?** Які файли зазвичай змінюються разом?
3.  **Який контекст?** Які патерни існують у схожих попередніх завданнях?
4.  **Який останній контекст?** Які області зараз активно розробляються?

Традиційні підходи (пошук за ключовими словами, статичний аналіз) не спрацьовують, тому що:
- Семантика коду є **композиційною** (значення виникає з комбінацій)
- Людська намір є **філософським** (ґрунтується на людському досвіді)
- Релевантність є **контекстуальною** (залежить від завдання, історії та стану проєкту)

### The Recommendation Quality Problem

При рекомендації змін коду, системи можуть зазнавати невдач у двох випадках:

| Режим невдачі | Опис | Наслідки |
|---|---|---|
| **Занадто консервативно** | Рекомендує лише добре відомі, часто змінювані файли | Пропускає нові, але релевантні зв'язки |
| **Занадто радикально** | Рекомендує багато нових зв'язків між модулями | Створює структуру "спагети" |

SIMARGL вирішує це через **антигігіанні метрики** (Новизна проти Структурності), які забезпечують баланс між інноваціями та архітектурною цілісністю.

## Research Pillars

### Pillar 1: Semantic Code Embeddings

Для розуміння коду потрібні різні стратегії вбудовування для різних рівнів деталізації та цілей.

#### 1.1 Compositional Code Embeddings

**Концепція**: Функції отримують значення з того, що вони викликають і використовують.

```
embedding(calculateTotal) = f(
    embedding(getPrice),
    embedding(applyDiscount),
    embedding(calculateTax)
)
```

**Основна інновація**: Векторні операції над семантикою коду
- `embedding(authenticateUser) - embedding(validateToken) + embedding(validateCookie) ≈ embedding(authenticateWithCookie)`

**Застосування**:
- Аналоговий пошук коду ("знайти функцію, схожу на X, але для Y")
- Семантичний diff ("що змінюється в значенні, а не в синтаксисі")
- Автозавершення коду з семантичною обізнаністю

#### 1.2 Cross-Layer Transformation Embeddings

**Концепція**: Відстежує, як дані протікають через архітектурні шари.

```
UI Layer → Service Layer → Repository Layer → Database
   ↓            ↓              ↓                ↓
UserDTO → UserService → UserRepository → users_table
```

**Основна інновація**: Вбудовування захоплюють ланцюжки трансформацій, а не лише окремі елементи.

**Застосування**:
- "Знайти весь код, що обробляє дані користувача від UI до бази даних"
- Аналіз впливу між архітектурними межами
- Підтримка рефакторингу для переструктурування шарів

#### 1.3 Keyword Entity Mapping

**Концепція**: Ключові слова є координатами в семантичному просторі.

Коли ми визначаємо "RULE" як релевантне, ми одночасно:
- **Включаємо**: RuleIndex, RuleUpdater, QualityProfileRules
- **Виключаємо**: Server, Plugin, DatabaseMigration (від'ємний простір)

**Основна інновація**: Контрастне навчання для обмежених семантичних регіонів.

### Pillar 2: Two-Phase Reflective Agent

Інтелектуальне навігація кодом потребує як дослідження, так і критичної оцінки.

#### Phase 1: Reasoning

- Аналіз опису завдання
- Пошук відповідних модулів, файлів і схожих завдань
- Генерація початкових рекомендацій
- Створення контексту з кількох джерел інформації

#### Phase 2: Reflection

- Критична оцінка результатів Phase 1
- Перевірка на відсутність контексту
- Перевірка логічної узгодженості
- Уточнення рекомендацій на основі більш глибокого аналізу

```
Task → [Phase 1: Explore & Reason] → Initial Results
                                          ↓
      [Phase 2: Reflect & Refine] ← Critique
                                          ↓
                                   Final Recommendations
```

**Основна інновація**: Самостійна система вдосконалення, натхненна людським експертним міркуванням.

---

### Pillar 3: Phenomenological Grounding

Код – це не лише синтаксис, а й відображення людських намірів, дій і значень.

#### 3.1 The Symbol Grounding Problem

LLM обробляють символи, не маючи зв'язку з живим досвідом:

| Концепція | Розуміння LLM | Людське розуміння |
|---|---|---|
| `setTimeout` | "Функція, що затримує виконання" | "Ті набридливе речі, що викликають збій" |
| `NullPointerException` | "Помилка при зверненні до null" | "Помилка, яка викликала збій у виробництві о 3 ранку" |

#### 3.2 Husserlian Framework

- **Наміреність**: Код завжди "про" щось (функціональність, потреба користувача)
- **Noema**: Артефакт коду, як він досвідчений (не лише текст)
- **Noesis**: Дія розуміння коду (читання з метою)

#### 3.3 Heideggerian Concepts

- **Zuhandenheit** (готовність до використання): Код, який "просто працює" – невидимі інструменти
- **Vorhandenheit** (наявність): Код, який стає видимим, коли зламано

#### 3.4 Code as Performative (Speech Act Theory)

Код не просто описує – він **виконує**:
- `deleteUser(id)` є **перлокуторною дією** з реальними наслідками
- Розуміння коду вимагає розуміння його **ефектів**, а не лише синтаксису

**Основна інновація**: Інтегрування феноменологічних понять в AI-підполег у навігації кодом.

---

### Pillar 4: Dual-Server RAG Architecture

Балансування між історичною мудрістю та поточним фокусом.

#### Historical Context Server

- **Покриття**: Всі попередні завдання (9799+)
- **Сила**: Патерни, що існують в різних проєктах, архітектурні рішення
- **Використання**: "Як раніше реалізовували автентифікацію?"

#### Recent Context Server

- **Покриття**: Останні 100 завдань
- **Сила**: Поточний фокус розробки, активні області
- **Використання**: "Що зараз активно розробляють, що може бути задіяне?"

#### Fusion Strategies

1.  **Weighted Fusion**: Конфігурований баланс (параметр α)
2.  **Reciprocal Rank Fusion**: Об'єднує рейтинги без нормалізації
3.  **Adaptive Fusion**: Вага, залежна від типу запиту

```
                    Query
                      ↓
        ┌─────────────┴─────────────┐
        ↓                           ↓
   Historical                    Recent
    Server                       Server
        ↓                           ↓
        └─────────────┬─────────────┘
                      ↓
               Fusion Layer
                      ↓
              Final Results
```

## SIMARGL Metrics Framework

### Core Antagonistic Metrics

#### Novelty@K

**Визначення**: Частка рекомендацій, які є **новими** (не в існуючому коді).

```
Novelty@K = |{r ∈ top-K : r ∉ ExistingRelations}| / K
```

- Висока (→1.0): Рекомендує переважно нові зв'язки (інноваційно)
- Низька (→0.0): Рекомендує переважно існуючі зв'язки (безпечно)

#### Structurality@K

**Визначення**: Частка рекомендацій, де обидва джерела та ціль належать до **одного й того самого модуля**.

```
Structurality@K = |{r ∈ top-K : module(r.source) == module(r.target)}| / K
```

- Висока (→1.0): Рекомендації залишаються в межах модулів (зберігають структуру)
- Низька (→0.0): Рекомендації перетинають межі модулів (можуть порушити структуру)

### The 2×2 Trade-off Matrix

```
                     Novelty
               LOW           HIGH
           ┌─────────────┬─────────────┐
      HIGH │ MAINTENANCE │  EVOLUTION  │
Struct.    │ Safe, no    │  Ideal:     │
           │ innovation  │  new + local│
           ├─────────────┼─────────────┤
      LOW  │ STAGNATION  │ DISRUPTION  │
           │ No value    │  Risky:     │
           │             │  new + cross│
           └─────────────┴─────────────┘
```

**Мета**: Максимізувати зону EVOLUTION (висока новизна + висока структурність)

### Data Flow

1.  **Task Input**: Опис завдання на природній мові
2.  **Keyword Extraction**: Визначення семантичних координат (RULE, SEARCH, тощо)
3.  **Embedding Generation**: Створення векторного представлення завдання на основі ключових слів
4.  **Dual-Server Search**: Пошук в обох серверах (історичному та останньому)
5.  **Fusion**: Об'єднання результатів з використанням відповідної стратегії
6.  **Agent Reasoning**: Фаза 1 дослідження, Фаза 2 рефлексії
7.  **Metric Evaluation**: Оцінка рекомендацій на основі Novelty/Structurality
8.  **Final Output**: Рейтингові рекомендації з поясненнями

## Evaluation Strategy

### Offline Evaluation

| Metric | Target | Опис |
|---|---|---|
| MAP@10 | >0.40 | Середнє значення |
| MRR | >0.50 | Середній реципрокний ранг |
| Recall@10 | >0.60 | Покриття релевантних файлів |
| Evolution@K | >0.35 | Нові + внутрішньо-модульні рекомендації |
| SES | >0.55 | Структурний бал еволюції |

### Online Evaluation

| Metric | Target | Опис |
|---|---|---|
| Task Completion | >75% | Розробники завершують завдання, використовуючи рекомендації |
| Time to First Relevant | <60s | Ефективність навігації |
| Context Sufficiency | >80% | Не потрібні додаткові пошуки |
