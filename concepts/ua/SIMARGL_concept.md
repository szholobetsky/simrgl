# SIMARGL: Метрики Структурної Цілісності для Адаптивного Навчання Графів Відношень

## Огляд Проєкту

**SIMARGL** — це дослідницький фреймворк для оцінки та балансування рекомендацій змін у програмному забезпеченні. Система рекомендує зв'язки між кодом (зв'язки між модулями, файлами чи функціями), які потрібно модифікувати разом при впровадженні нових вимог.

Ключова інновація — **пара антагоністичних метрик**, які вимірюють компроміс між:
- **Новизною (Novelty)** — рекомендування НОВИХ зв'язків (інновація)
- **Структурністю (Structurality)** — збереження ІСНУЮЧОЇ модульної структури (стабільність)

### Етимологія

Назва походить від Симаргла (Simargl) — слов'янського крилатого божества у формі пса, який охороняв Дерево Життя та передавав повідомлення між богами та смертними. Його довгий хвіст іноді заплутувався, спричиняючи неправильну доставку повідомлень — метафора компромісу між доставкою нової інформації (новизна) та підтримкою структурного порядку (цілісність).

## Постановка Проблеми

Коли рекомендаційна система пропонує зміни коду, вона може допустити два типи помилок:

1. **Занадто консервативна** — рекомендує лише існуючі, добре відомі зв'язки
   - Висока точність, але без цінності (проблема "зупиненого годинника")
   - Система стагнує, немає інновацій

2. **Занадто руйнівна** — рекомендує багато нових міжмодульних зв'язків
   - Висока новизна, але руйнує модульну структуру
   - Створює архітектуру "спагеті-коду"

**Мета**: Знайти рекомендації, які є ОДНОЧАСНО новими ТА поважають існуючу модульну структуру.

---

## Основні Метрики

### Novelty@K (Новизна@K)

**Визначення**: Частка рекомендованих зв'язків, які є НОВИМИ (відсутніми у існуючій кодовій базі).

```
Novelty@K = |{r ∈ top-K : r ∉ ExistingRelations}| / K
```

- **Високе значення (→1.0)**: Система рекомендує переважно нові зв'язки
- **Низьке значення (→0.0)**: Система рекомендує переважно існуючі зв'язки

**Інтерпретація**:
- Novelty@K = 1.0 означає всі рекомендації нові (можливо ризиковано)
- Novelty@K = 0.0 означає всі рекомендації вже існують (без доданої цінності)

---

### Structurality@K (Структурність@K)

**Визначення**: Частка рекомендованих зв'язків, де ОБИДВА джерело та ціль належать до ОДНОГО модуля.

```
Structurality@K = |{r ∈ top-K : module(r.source) == module(r.target)}| / K
```

- **Високе значення (→1.0)**: Рекомендації залишаються в межах модулів
- **Низьке значення (→0.0)**: Рекомендації перетинають межі модулів

**Інтерпретація**:
- Structurality@K = 1.0 означає всі рекомендації внутрішньомодульні (зберігає структуру)
- Structurality@K = 0.0 означає всі рекомендації міжмодульні (може порушити структуру)

## Матриця Компромісів 2×2

Комбінування Новизни та Структурності створює чотири зони:

```
                          Novelty@K
                     (нові зв'язки)
                          ВИСОКА (1.0)
                             ▲
                             │
        ┌────────────────────┼────────────────────┐
        │                    │                    │
        │     РУЙНУВАННЯ     │     ЕВОЛЮЦІЯ       │
        │                    │                    │
        │   Нові + Між-      │   Нові + Внутрішньо│
        │   модульні зв'язки │   модульні зв'язки │
        │                    │                    │
        │   ⚠️ РИЗИКОВАНО    │   ✅ ІДЕАЛЬНО      │
        │                    │                    │
НИЗЬКА◄─┼────────────────────┼────────────────────┼───► ВИСОКА
(0.0)   │                    │                    │    (1.0)
        │    СТАГНАЦІЯ       │    ОБСЛУГОВУВАННЯ  │  Structurality@K
        │                    │                    │  (в межах модулів)
        │   Старі + Між-     │   Старі + Внутрішньо│
        │   модульні зв'язки │   модульні зв'язки │
        │                    │                    │
        │   ❌ БЕЗКОРИСНІ    │   🔧 БЕЗПЕЧНО      │
        │                    │                    │
        └────────────────────┼────────────────────┘
                             │
                             ▼
                          НИЗЬКА (0.0)
```

### Опис Зон

| Зона | Новизна | Структурність | Опис | Дія |
|------|---------|---------------|------|-----|
| **ЕВОЛЮЦІЯ** | Висока | Висока | Нові зв'язки, що поважають межі модулів | ✅ Ідеальний стан — прагнути до цього |
| **РУЙНУВАННЯ** | Висока | Низька | Нові зв'язки, що перетинають межі модулів | ⚠️ Може порушити архітектуру |
| **ОБСЛУГОВУВАННЯ** | Низька | Висока | Існуючі зв'язки в межах модулів | 🔧 Безпечно, але без інновацій |
| **СТАГНАЦІЯ** | Низька | Низька | Існуючі міжмодульні зв'язки | ❌ Без цінності, уникати |

## Похідні Метрики

### Метрики Конкретних Зон

Кожна рекомендація потрапляє рівно в одну зону:

```python
Evolution@K    = |{r ∈ top-K : r є НОВИМ та r є ВНУТРІШНЬО-модульним}| / K
Disruption@K   = |{r ∈ top-K : r є НОВИМ та r є МІЖ-модульним}| / K
Maintenance@K  = |{r ∈ top-K : r є СТАРИМ та r є ВНУТРІШНЬО-модульним}| / K
Stagnation@K   = |{r ∈ top-K : r є СТАРИМ та r є МІЖ-модульним}| / K
```

**Обмеження**: Evolution@K + Disruption@K + Maintenance@K + Stagnation@K = 1.0

### Композитні Метрики

**Оцінка Структурної Еволюції (SES)**:
```python
SES = sqrt(Novelty@K × Structurality@K)
```
- Діапазон: [0, 1]
- Штрафує, якщо будь-яка метрика низька
- Максимум, коли обидві високі

**Гармонійна Оцінка Еволюції (HES)**:
```python
HES = 2 × Novelty@K × Structurality@K / (Novelty@K + Structurality@K)
```
- Більш чутлива до дисбалансу, ніж SES

## Чому Баланс Важливий

### Проблема Упередженості до Популярності

У реальних кодових базах кілька "ключових" модулів змінюються дуже часто:

```
Розподіл змін модулів:
  CoreUtils (45%)  ████████████████████████████████████
  DataLayer (25%)  ████████████████████
  UI (15%)         ████████████
  Auth (10%)       ████████
  Config (5%)      ████
```

Наївна система навчається ЗАВЖДИ рекомендувати CoreUtils, тому що:
- Він з'являється в 45% усіх змін
- Рекомендування дає 45% "точності"
- Але це **проблема "зупиненого годинника"** — іноді правильно, але безкорисно

### Проблема Спагеті-Коду

Система, що оптимізує лише новизну, може рекомендувати:

```
Завдання: "Додати автентифікацію користувача"
Рекомендації:
  1. PaymentService → AuthController  (НОВЕ, міжмодульне)
  2. ReportGenerator → UserModel      (НОВЕ, міжмодульне)
  3. CacheManager → SessionService    (НОВЕ, міжмодульне)
```

Усі нові зв'язки! Висока новизна! Але це створює тісне зв'язування між непов'язаними модулями.

### Ідеальний Результат

Збалансована система рекомендує:

```
Завдання: "Додати автентифікацію користувача"
Рекомендації:
  1. AuthController → UserValidator   (НОВЕ, той самий модуль auth) ✅
  2. AuthController → TokenService    (НОВЕ, той самий модуль auth) ✅
  3. UserService → AuthController     (НОВЕ, міжмодульне але логічне) ⚠️
```

- Нові зв'язки (новизна)
- Переважно в межах модуля auth (структурність)
- Міжмодульний зв'язок лише там, де семантично необхідно

## Рекомендації до Впровадження

### Обчислення Метрик

```python
class SIMARGLMetrics:
    def __init__(self, existing_relations: Set[Tuple], module_map: Dict[str, str]):
        """
        existing_relations: Множина кортежів (джерело, ціль, тип_зв'язку)
        module_map: Словник відображення ID функції/файлу на ID модуля
        """
        self.existing = existing_relations
        self.modules = module_map

    def is_novel(self, relation) -> bool:
        return (relation.source, relation.target, relation.type) not in self.existing

    def is_intra_module(self, relation) -> bool:
        src_module = self.modules.get(relation.source)
        tgt_module = self.modules.get(relation.target)
        return src_module is not None and src_module == tgt_module

    def compute_metrics(self, recommendations: List, k: int) -> Dict:
        top_k = recommendations[:k]

        novel = [r for r in top_k if self.is_novel(r)]
        intra = [r for r in top_k if self.is_intra_module(r)]

        novelty = len(novel) / k
        structurality = len(intra) / k

        evolution = len([r for r in novel if self.is_intra_module(r)]) / k
        disruption = len([r for r in novel if not self.is_intra_module(r)]) / k
        maintenance = len([r for r in top_k if not self.is_novel(r) and self.is_intra_module(r)]) / k
        stagnation = len([r for r in top_k if not self.is_novel(r) and not self.is_intra_module(r)]) / k

        ses = (novelty * structurality) ** 0.5

        return {
            'Novelty@K': novelty,
            'Structurality@K': structurality,
            'Evolution@K': evolution,
            'Disruption@K': disruption,
            'Maintenance@K': maintenance,
            'Stagnation@K': stagnation,
            'SES': ses
        }
```

### Балансування на Основі Температури

Використовуйте параметр температури для контролю балансу новизна-структурність:

```python
def balanced_score(semantic_score, novelty_score, structural_score, temperature=1.0):
    """
    temperature < 1.0: Віддавати перевагу структурі (консервативно)
    temperature = 1.0: Збалансовано
    temperature > 1.0: Віддавати перевагу новизні (дослідницьки)
    """
    novelty_weight = temperature / (1 + temperature)
    structure_weight = 1 / (1 + temperature)

    combined = (
        0.5 * semantic_score +
        novelty_weight * novelty_score +
        structure_weight * structural_score
    )
    return combined
```

## Науковий Внесок

### Що Нового в SIMARGL

1. **Метрика Structurality@K** — вимірює вплив рекомендацій на модульну структуру
2. **Пара антагоністичних метрик** — Новизна vs Структурність як протилежні сили
3. **Декомпозиція 2×2** — зони Еволюція/Руйнування/Обслуговування/Стагнація
4. **Застосування до рекомендацій змін коду** — перенесення з рекомендаційних систем на еволюцію ПЗ

### Ключова Ідея

Існуючі дослідження рекомендаційних систем фокусуються на **задоволенні користувача** (новизна vs точність).
SIMARGL фокусується на **здоров'ї системи** (новизна vs структурна цілісність).

Це нова перспектива, що об'єднує:
- Теорію рекомендаційних систем
- Принципи архітектури ПЗ
- Дослідження еволюції ПЗ

## Глосарій

| Термін | Визначення |
|--------|------------|
| **Зв'язок (Relation)** | Направлене з'єднання між двома елементами коду (виклики функцій, імпорти, потік даних) |
| **Модуль (Module)** | Згуртована одиниця коду (пакет, простір імен, директорія) |
| **Внутрішньомодульний (Intra-module)** | Зв'язок, де джерело та ціль в одному модулі |
| **Міжмодульний (Inter-module)** | Зв'язок, де джерело та ціль в різних модулях |
| **Новизна (Novelty)** | Чи є зв'язок новим (відсутнім у поточній кодовій базі) |
| **Структурність (Structurality)** | Чи поважає зв'язок межі модулів |
| **Еволюція (Evolution)** | Новий + внутрішньомодульний (ідеально) |
| **Руйнування (Disruption)** | Новий + міжмодульний (ризиковано) |
| **Температура (Temperature)** | Параметр, що контролює баланс новизна-структура |
| **SES** | Structural Evolution Score — композитна метрика |
