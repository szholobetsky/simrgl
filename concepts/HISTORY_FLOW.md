# HISTORY_FLOW — концептуальний blame для 1bcoder

## Задача

`git blame` відповідає на питання **"хто востаннє чіпав цей конкретний рядок"** — синтаксично, по одному рядку, без жодного знання про сенс. `/flow history` відповідає на інше питання: **"як еволюціонувала ця бізнес-концепція по всій системі за весь час"** — крізь файли, крізь задачі, крізь час. Це не заміна git blame (для одного конкретного рядка він і лишається правильним інструментом), а щось на рівень вище — концептуальний blame замість синтаксичного.

Практична цінність: коли приходить legacy-проєкт на підтримку, `git blame`/`grep` безсилі, якщо не знаєш точних слів у коді. Стаття "ROUND" з накопиченою історією ("задача ABC-123 замінила ROUND на CEIL") дозволяє швидко зрозуміти, як історично вирішувалась конкретна предметна проблема, не знаючи наперед усіх ідентифікаторів, які до неї причетні.

## Чому не `/flow glossary`

`/flow glossary` (див. `GLOSSARY.md`) — це корпус-незалежна екстракція TERMS→DEFINITION→FACTS, де кожен prose-чанк аналізується **незалежно** від сусідніх. Це працює для документації, де кожен фрагмент тексту самодостатній.

Задачі+дифи — **не самодостатня проза**. Демонстрація на реальних даних (`sqlfluff`, задача "Merging arguments to exclude rules", `joined/24.txt`): diff містить ~230 рядків, з яких лише ~15-20 стосуються суті (заміна `exclude_rule_string`/`rule_blacklist`), решта — видалення порожніх рядків CHANGELOG.md і заміна `_italic_`→`*italic*` в DOCS.md. Якби модель почанкувала цей diff на 3 шматки без збереження контексту задачі, до другого чанку вона забула б, про що взагалі йдеться, і залишилась би з голими `get_linter`/`rule_whitelist` без жодного натяку на "виключення правил". Гірше — сам напрямок diff тут оманливий: рядки з `-` перед `exclude_rule_string` виглядають як видалення фічі, хоча насправді це артефакт merge гілки `ac/clearup`, створеної до появи фічі на main.

`/flow history` — task-first, структурно чанкований (по межах `FILE:`/`@@`, не по символах), з постійно "теплим" контекстом задачі, і пише в нову секцію `HISTORY:`, а не в `FACTS:` — бо **факти — то факти, а історія — то історія** (статичні властивості поняття проти часових записів про зміни).

## Джерело даних

`simargl export --all` (окремо `tasks/`, `commits/`) — **не** `--join`. Зв'язування задача↔коміт відбувається всередині самого flow під час обробки, а не наперед зашите в один файл.

## Процедура — один цикл, тільки по задачах

Реєстр: `task_id → done`, живе поруч із глосарієм (`_history/`, за аналогією з `_think/`). Одиниця "роботи вже зроблено" — вся задача цілком, не пара (термін, задача): один прохід задачі одразу закриває всі терміни, яких вона стосується.

Для кожної необробленої задачі:

1. Читаємо TITLE+DESCRIPTION.
2. LLM виділяє кандидатів у ключові слова (той самий `TERMS:`-промпт, що вже є в `glossary.py`).
3. Fuzzy-пошук кандидатів проти існуючих статей глосарія — **перевикористовуємо `glossary.py`'s `extract(query_text, terms)` напряму**, нового коду для матчингу не треба.
4. Кандидати без існуючої статті — нову статтю не створюємо (недостатньо даних ні для `DEFINITION:`, ні для `FACTS:`), але їхню історію все одно приписуємо до **найважливішої з існуючих**, що збіглася, а не відкидаємо.
5. Ріжемо diff задачі на чанки по межах `FILE:` і `@@` (файл → git hunk; hunk уже несе clas/def-контекст завдяки власній конвенції git `@@ ... @@ <enclosing function>`).
6. У межах одного чанку — **один спільний кешований префікс** (система + чанк + "ознайомся"-інструкція), а кожне наступне питання про конкретний термін — незалежна коротка "гілка" від цієї самої точки (відповідь на попередній термін **не** дописується в контекст наступного запиту). Це максимізує влучання в KV-cache і не забруднює аналіз одного терміна рештками міркувань про інший (див. розділ нижче).
   **Промпт обов'язково явно пояснює семантику diff**: рядок з `-` — видалення коду, з `+` — додавання. Без цього мала локальна модель може переплутати напрямок зміни (точно як у прикладі з `joined/24.txt` вище).
7. Відповідь LLM дописується в `HISTORY:` відповідної статті, з тегом `[file: ...]` — та сама конвенція, що й `FACTS:`/`DEFINITION:` у `glossary.py`.
8. Задача позначається опрацьованою в реєстрі — назавжди, один прохід, незалежно від того, скількох термінів вона торкнулась.

**`--loop term` (обхід термін-перший, потім задачі) свідомо відкинутий**, а не об'єднаний з task-first в один двигун із двома стратегіями відбору. Причина — KV-cache: term-outer цикл змушує повторно "холодно" перечитувати той самий чанк для кожного терміна, який у ньому згадується, натомість task-outer цикл читає кожен чанк рівно один раз і питає про всі релевантні терміни, поки він ще "теплий" у кеші.

## KV-cache-ефективний виконавчий цикл

Кешується не "повідомлення" в абстрактному сенсі розмови, а **буквальний префікс токенів**. Якщо запит 2 (`[чанк]+"ознайомся"+"перевір ROUND"`) починається рівно з тих самих токенів, що й запит 1 (`[чанк]+"ознайомся"`), рушій із підтримкою prefix-caching перерахує лише хвіст, що розходиться.

Схема: **не** нарощувати одну розмову (запит1→відповідь1→запит2→відповідь2→...), а щоразу відгалужуватись від тієї самої фіксованої точки, ігноруючи попередні відповіді. Це і максимізує влучання в кеш (спільний префікс лишається мінімальним і незмінним), і уникає змішування контексту різних термінів у відповіді моделі.

**Емпірично підтверджено (2026-07-22)**, на двох хостах/моделях (`panteon.local:11434`, `gemma3:4b-it-q4_K_M`; `localhost:11434`, `gemma3:1b`; через Ollama `/api/generate`, chunk ≈ 2130 токенів):

| Запит | prompt_eval_count | час prompt_eval (panteon) | час prompt_eval (localhost) |
|---|---|---|---|
| холодний виклик (chunk+"ознайомся") | 2130 | 63.1 с | 17.2 с |
| той самий префікс + "ROUND", **незалежний** виклик | 2145 | 0.95 с | 0.22 с |
| той самий префікс + "CEIL", **незалежний** виклик | 2146 | 0.53 с | 0.12 с |
| контроль: інший chunk такого ж розміру, без спільного префіксу | 1893 | 56.1 с | — |

Ollama (через llama.cpp) **автоматично** розпізнає спільний префікс між **незалежними** викликами `/api/generate` — жодного `context`-параметра передавати не потрібно. Прискорення prompt_eval — 60-130×, і воно повторюється для другого незалежного відгалуження від тієї самої точки (не лише одноразовий збіг з попереднім викликом). Контрольний запит (інший chunk, без спільного префіксу) підтверджує, що ефект саме від кешування префіксу, а не від загального "прогріву" моделі.

Порожній "ознайомся"-виклик (відповідь якого одразу викидається) можна прибрати — перший реальний запит одразу може бути `[чанк]+"ознайомся"+"перевір ROUND"` без окремого round-trip'у, кеш все одно наповнюється з першого ж виклику.

`_llm()` у поточному `glossary.py` — одноразовий виклик (новий `messages`-список щоразу); для `/flow history` потрібен режим, що вміє **гілкуватися** від фіксованого спільного префіксу, а не одноразовий виклик і не одна зростаюча розмова.

## Файлова структура

```
.1bcoder/glossary/<project>/
  <term>.md                 DEFINITION: / FACTS: / LINK: / HISTORY:   ← нова секція
  _history/
    processed_tasks.yaml    task_id → done (реєстр)
```

`HISTORY:` — append-only список, як `FACTS:`, але семантично окремий розділ (не змішується з фактами про статичні властивості поняття). Пошук працює так само, як і для інших секцій — звичайний підрядковий пошук по всьому тексту статті, жодної спеціальної логіки не треба.

## Command surface (орієнтовно, ще не реалізовано)

```
/flow history index --glossary <path> --tasks <path>
```

Синтаксис і назви прапорців можуть змінитись при реалізації.

## Що свідомо не робимо (MVP)

- **`--loop term`** — відкинуто (KV-cache аргумент вище).
- **Відкриття нових термінів** — `/flow history` тільки збагачує вже існуючі статті глосарія, не створює нові (це робота `/flow glossary index`).
- **Реєстр по парі (термін, задача)** — виявився зайвим; реєстр лише по задачі.
- **Cross-glossary parent/reference definition lookup** (ідея "показати системі батьківський глосарій і брати з нього визначення") — обговорювалась окремо, лишається відкритим питанням, не входить у цей MVP.
- **Читання `joined/`-виводу simargl export** — свідомо використовуємо `--all` (окремо tasks/commits), не `--join`.
