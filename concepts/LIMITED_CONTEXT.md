# LIMITED_CONTEXT — Ковзаюче вікно контексту для локальних LLM

## Задача

`/flow ctxtimer` (див. [`CTXTIMER.md`](CTXTIMER.md)) емпірично показує: для багатьох локальних моделей на CPU-only чи слабкому iGPU заліза (Пантеон, [`IGPU_OFFLOAD_THRESHOLD.md`](IGPU_OFFLOAD_THRESHOLD.md)) існує жорстка стеля контексту — 3-8K токенів залежно від моделі/архітектури (dense vs MoE coverage saturation). Після цієї стелі KV-cache вже не допомагає: модель або не встигає почати генерацію до timeout, або генерація стає настільки повільною, що продовжувати розмову практично неможливо.

**Наслідок**: якщо KV-cache перестає давати перевагу після N токенів, то *немає сенсу* намагатись утримати весь накопичений контекст (розмова може вирости до "розміру Міннесоти" — 20K, 50K токенів). Натомість треба явно **обрізати** контекст до розміру, який модель реально тягне, і придумати дешевий спосіб не втратити повністю сенс відрізаного "хвоста".

Це paralельна, але окрема задача від ctxtimer: ctxtimer *вимірює* межу; LIMITED_CONTEXT *живе* в межах цієї виміряної межі, зберігаючи розмову практично нескінченно довгою для користувача, ціною втрати частини деталей із середини.

## Синтаксис команди (зростаюча складність)

```
/ctx window last:N              # тільки останні N токенів контексту, більше нічого
/ctx window off                 # вимкнути — повертаємось до повного контексту

/ctx window first:M last:N      # фіксований початок (M токенів) + ковзаючий кінець (N токенів)
                                 # усе між ними відкидається

/ctx window first:M last:N mid:rs limit:K
                                 # + додатково до K токенів із середини, обраних Reservoir Sampling

/ctx window first:M last:N mid:bm25 limit:K
/ctx window first:M last:N mid:dp   limit:K
/ctx window first:M last:N mid:tr   limit:K
                                 # той самий mid-слот, інший алгоритм вибору (BM25 / DP / TextRank)
```

Загальний бюджет контексту, що фактично йде в LLM: `first (M) + mid (≤K) + last (N)` токенів, замість повного накопиченого контексту.

**Чому `first:M` фіксований**: перші повідомлення розмови зазвичай містять початкову мету/постановку задачі — те, "з чого все почалось". Це найцінніша інформація для довгої розмови, і вона *не* повинна конкурувати за місце з рештою — тому фіксований (не ковзаючий) блок.

**Чому `last:N` ковзаючий**: це те, що модель бачить у форматі, схожому на звичайну розмову — свіжий, послідовний, зв'язний контекст безпосередньо перед поточним повідомленням.

**Роль `mid`**: усе, що випадає між `first` і `last` — не обов'язково зникає безслідно. `mid:<algo> limit:K` дозволяє додати до K токенів "конспекту" середини, обраного одним із чотирьох алгоритмів нижче.

## Алгоритми для `mid`

Гранулярність відбору — цілі повідомлення за замовчуванням (зберігає когерентність; репліка не розривається на половину). Для TR/DP можлива додаткова гранулярність — окремі речення всередині повідомлень, якщо потрібна точніша упаковка в бюджет `limit:K`.

### `mid:rs` — Reservoir Sampling

Найдешевший, "тупий, але чесний" варіант. Класичний reservoir sampling по потоку "середніх" повідомлень — рівномірний випадковий семпл K токенів (в одиницях повідомлень, доки не перевищено `limit`), без збереження всього масиву в пам'яті одночасно. **Не query-conditioned** — не враховує, що зараз обговорюється; просто репрезентативний зріз середини розмови. Швидкість: O(n), без сторонніх залежностей.

### `mid:bm25` — BM25 retrieval, query = поточне повідомлення

Індексувати кожне "середнє" повідомлення як документ у in-memory SQLite FTS5-таблиці, і ранжувати проти термінів з **поточного** (щойно відправленого) повідомлення користувача — тобто query-conditioned retrieval: підтягуються саме ті старі факти з середини розмови, які релевантні тому, що обговорюється зараз.

**Готовий код для перевикористання**: `_fts_rank(terms, file_contents, top_k)` у `C:\Project\1bcoder\chat.py:167-188` вже реалізує BM25 через in-memory SQLite FTS5 (зараз ранжує вміст файлів для RAG-пошуку по коду). Той самий движок підходить майже без змін — `file_contents` замінюється на `{message_id: message_text}` зі "середніх" повідомлень.

Після відбору топ-K за BM25-score — **пересортувати за хронологічним порядком** (позицією в оригінальній розмові), щоб вставка в контекст читалась як зв'язний, а не перемішаний текст.

### `mid:dp` — Dynamic Programming (budgeted knapsack)

Кожному повідомленню/реченню призначається:
- **вага** = кількість токенів (обмеження бюджету `limit:K`)
- **цінність** = BM25-score відносно поточного query, або TF-IDF novelty score, або комбінація

Розв'язується 0/1 knapsack через DP: максимізувати сумарну цінність при обмеженні на сумарну вагу ≤ K. Точний розв'язок дешевий, бо кількість кандидатів (повідомлень у середині розмови) зазвичай мала (десятки-сотні, не мільйони) — O(n·K) за часом і пам'яттю, цілком прийнятно.

Відмінність від BM25 top-K: BM25 просто бере топ-K за score, ігноруючи, що бюджет токенів можна заповнити щільніше (наприклад, два коротших релевантних повідомлення замість одного довгого). DP дає оптимальну упаковку бюджету, а не просто топ за релевантністю.

### `mid:tr` — TextRank / LexRank (графове ранжування)

Побудувати граф схожості між реченнями/повідомленнями середини (косинус по TF-IDF векторах, або простий word-overlap/Jaccard), прогнати PageRank-подібний алгоритм по графу, взяти топ-K за центральністю в межах бюджету. На відміну від BM25, **не query-conditioned** — оцінює "структурну важливість" речення в межах самої розмови (те, на що посилаються/перегукуються інші речення), а не релевантність до поточного питання. Класична техніка екстрактивного резюмування, без LLM.

## LLM-based резюме (окрема гілка, не "mid", а альтернатива компактизації)

1bcoder вже має `/ctx compact` і, що важливо, `/ctx compact profile: <name>` — резюмування **зовнішньою** моделлю (через profile), а не основною Ollama-моделлю поточної розмови. Це напряму вирішує вимогу "резюмування має робити окрема легка модель, щоб не страждав KV-cache і VRAM основної моделі" — механізм уже існує, просто вимагає використання разом із `/ctx window`, не замість нього.

Компроміс тут — рекурсивна втрата деталей при резюме-на-резюме — визнаний і прийнятий: аналогія з людською пам'яттю (gist замість verbatim, "розпитати" деталі через RAG/simargl-index при потребі) — це усвідомлений trade-off, не помилка.

## План реалізації

### Фаза 1 — 1bcoder (спочатку тут, для експериментів)

Нова підкоманда `/ctx window ...` у `_cmd_ctx` (`chat.py:4067`), поруч із вже існуючими `cut`/`save`/`load`/`savepoint`/`compact` — той самий dispatch-патерн (`if parts[1] == "window": ...`). Операція діє безпосередньо на `self.messages` перед відправкою в `_stream_chat()` — тобто не змінює накопичену історію назавжди (на відміну від `/ctx cut`, яка видаляє повідомлення), а лише **проектує** урізаний/змішаний варіант на кожен виклик LLM, поки `window` увімкнено.

Реалізувати всі 4 варіанти команди й усі 3 mid-алгоритми (RS, BM25, DP, TR) в 1bcoder першим — щоб мати пісочницю для порівняння методів на реальних довгих розмовах, перш ніж переносити в vyrii.

### Фаза 2 — vyrii (централізовано в одному місці)

На відміну від `ctxtimer` (де порт у vyrii вимагав повністю окремого модуля через різницю в error-handling конвенціях, див. [`project_ctxtimer` пам'ять]), тут природний дім — **`vyrii/engine.py`**, оскільки:
- Усі три UI vyrii (Flask, FastAPI, Gradio) вже йдуть через один і той самий `engine.stream_chat()`/`complete()`.
- Вікно/mid-алгоритми — це препроцесинг списку `messages` **перед** викликом `stream_chat()`, не залежить від per-UI специфіки (SSE, Gradio generators тощо), на відміну від ctxtimer, де сама відповідь-timeout логіка була сплетена з per-backend error handling.

План: одна функція в `engine.py` (чи новий сусідній модуль `vyrii/ctxwindow.py`, за аналогією з `vyrii/ctxtimer.py`) — `apply_window(messages, mode, first, last, mid_algo, limit) -> list[dict]`, що викликається один раз у кожному з трьох бекендів безпосередньо перед `stream_chat()`/`complete()`, керована або per-request параметром, або збереженою в `~/.vyrii/config.json` настройкою (за зразком `ollama_kv_cache`/`ollama_vulkan` — той самий `_read_cfg()`/`_write_cfg()` механізм).

BM25-частина в vyrii використовує той самий підхід через `sqlite3` in-memory FTS5 (Python stdlib, без нових залежностей) — без прив'язки до `_fts_rank` 1bcoder (окрема копія логіки, як і з ctxtimer — 1bcoder і vyrii лишаються незалежними пакетами).

### Фаза 3 — Дослідження: що працює краще

Ідея користувача: "реалізувати все, а потім дослідити, які методи/комбінації працюють краще" — це не суперечить академічній цінності: тут вимірюється не швидкість (як у ctxtimer), а **якість утримання сенсу розмови**. Пропонований протокол оцінки, за аналогією з MAP/MRR-методологією з exp2/exp3:

1. Взяти довгу реальну (чи синтетично згенеровану) розмову з фактами, розкиданими по "середині".
2. Поставити запитання, що вимагають пригадати факт із середини розмови.
3. Порівняти відповідь моделі при: (a) повному контексті (ground truth, якщо влазить), (b) `last:N` тільки, (c) `first:M last:N` без mid, (d) кожен із mid-алгоритмів.
4. Метрика: чи відповідь містить потрібний факт (можна LLM-as-judge чи проста keyword-перевірка) — аналог precision/recall retrieval-оцінки, тільки застосований до розмовної пам'яті, а не до пошуку коду.

Це може стати окремим прикладним результатом (не основна лінія дисертації — task-to-code retrieval — але суміжний, citable внесок про практичне управління контекстом у продуктовій екосистемі SIMARGL).
