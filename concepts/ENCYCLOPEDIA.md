# ENCYCLOPEDIA — необмежена стаття на відміну від компактного glossary

## Задача

`/flow glossary` (див. [`GLOSSARY.md`](GLOSSARY.md)) навмисно компактний: `FACTS:` — атомарні однорядкові твердження, `DEFINITION:` — кілька речень. Це не стилістичний вибір, а прямий наслідок цільової аудиторії — малої локальної моделі з бюджетом контексту 3-8К токенів. Реальний прогін (sqlfluff docs, gemma) показав ціну цього компромісу: багатий контекст (код `generate_sidebar_config`, конкретна структура `{"text":..., "link":...}`, порядок сортування) стискається до 4 однорядкових фактів. Для швидкого пошуку це нормально. Для розуміння — ні.

## Аналогія: БЕС проти БСЕ

**Glossary — Большой Энциклопедический Словарь**: один том, компактні статті, для швидкої довідки — "що таке амальгама" за 10 секунд.

**Encyclopedia — Большая Советская Энциклопедия**: багатотомне видання, статті без обмеження розміру, для глибокого занурення — "написати реферат про амальгаму".

Обидва інструменти потрібні одночасно і не заміняють один одного. Якщо коротких фактів достатньо для відповіді — залишаєшся в glossary. Якщо ні — йдеш у encyclopedia. Розмір статті в encyclopedia свідомо не обмежується: якщо побудова забирає тиждень на окремому сервері — це прийнятна ціна.

## Прерогатива: provenance в самому glossary — те, чого зараз бракує

**Проблема виявлена emпірично, не гіпотетично**: жоден факт чи визначення в поточному `term.md` не зберігає, з якого чанка/файлу він узятий. `_write_term_file` (glossary.py:200-217) пише лише `DEFINITION:`/`FACTS:`/`LINK:` — жодного поля джерела.

Це не дрібниця для encyclopedia з двох причин:
1. Без provenance неможливо процитувати оригінальний уривок дослівно (підхід 1 нижче) — треба заново шукати, звідки взявся факт.
2. Без provenance зв'язок терм → задачі/коміти (розділ "Change History" нижче) вимушено йде через RAG-здогадку замість детермінованого lookup — і саме тут ховається ризик "гарбуза".

**Необхідне вдосконалення glossary (до encyclopedia)**: кожен факт і кожне визначення повинні супроводжуватись джерелом:

```
FACTS:
- The list `items` accumulates config entries for sidebar links.
  [src: docs/generate.py:42-47]
```

Для фактів, похідних від задач/комітів (коли glossary/encyclopedia починає індексувати ще й SIMARGL-датасети задач):

```
- Fixed line_position handling for nested WHERE clauses.
  [task: SQLFLUFF-1234] [commit: abc123f] [src: src/sqlfluff/core/layout.py:88-101]
```

Формат джерела: шлях файлу **на момент індексації** (може розійтися з поточним HEAD, але це і є цінність — фіксує стан, при якому факт був правдивим), рядки від-до, і за наявності — номер задачі й коміту. Це найменша, найважливіша зміна, потрібна перед encyclopedia — без неї весь підхід 1 (нижче) і секція Change History не мають опори.

## Два підходи до побудови статті

### Підхід 1 — той самий алгоритм, що й у glossary (запропоновано користувачем)

Чанкуємо файл так само, як і glossary. Але замість "витягни атомарні факти" — питаємо модель "допиши частину статті, яку можна побудувати з цього чанка". Наступний чанк — наступна частина. Накопичення відбувається так само інкрементально, як `FACTS:` зараз, тільки замість однорядкових тверджень — абзаци.

Ключове спостереження користувача: документація вже **кристалізована** — це вже добре написана проза. Для прозових джерел не обов'язково просити модель перефразовувати/стискати — можна **копіювати оригінальний текст** (за індексом термінів, які вже відомі з glossary), не втрачаючи жодної деталі. LLM тут потрібна лише для розбиття/прив'язки тексту до правильного терма, не для стиснення змісту.

**Переваги**: нуль втрати інформації для прозових джерел; той самий, вже перевірений і зрозумілий алгоритм, що й glossary; не потребує LLM-виклику для копіювання (тільки для рішення "чи стосується цей чанк цього терма" і, можливо, легкого редагування шва між чанками).

**Обмеження**: для коду (не кристалізованого, не готової прози) цей підхід повертається до того самого стиснення, від якого ми тікаємо — копіювати код можна дослівно (він і так точний), але "написати статтю про те, що робить код" все одно вимагає синтезу, як і в glossary.

### Підхід 2 — оркестрація через `/deepagent_md` по одному терму (запропоновано як альтернатива)

Замість нового рушія — виклик вже існуючого `/deepagent_md` окремо для кожного терма зі списку glossary:

```
для кожного term у glossary.md:
  /deepagent_md
    target:  "<term>: <коротке визначення з glossary як якір>"
    plan:    overview, details          # 1-2 рівні, НЕ 3 — узгоджено з користувачем
    rag:     <project що вказує на simargl-індекс над тим самим корпусом>
    file:    encyclopedia/<project>/<term>/article.md
```

**Переваги**: майже нуль нового коду — вся важка машинерія (parallel workers, tree building, concrete-leaf detection, Y/n/q retry, які ми вже доробили цієї сесії) вже існує і перевикористовується напряму.

**RAG-grounding вже реалізований**: `deepagent_md.py:320-380` (`_rag_research`) вже викликає `simargl.searcher.search(query, mode="file", ...)` і підмішує знайдені файли як контекст. Твоя ідея "передати термін або факт як запит, нехай RAG сама знайде звідки він" — це вже параметр `--rag <project>`, не нова фіча.

**Ризик retrieval для цього сценарію ще не виміряний.** `mode="file"` — файловий retrieval над корпусом документації; наскільки надійно він знаходить правильне джерело для терму/факту як запиту — окреме емпіричне питання, яке ще не досліджувалось (це інший domain і інший тип запиту, ніж будь-що вже виміряне в exp0-exp3, тож перевикористовувати ті цифри тут не можна). Перш ніж покладатись на цей шлях в encyclopedia — потрібна власна перевірка на реальному корпусі.

### Який підхід використовувати

Не взаємовиключні: **підхід 1 для прозових джерел** (документація, вже кристалізований текст — нуль втрати, дешево), **підхід 2 для коду й ситуацій, де потрібен синтез, а не копіювання** (коли джерело не є вже готовою статтею). Обидва пишуть у ту саму файлову структуру `encyclopedia/<project>/<term>/`.

## Change History — задачі й коміти в кінці статті

Ідея користувача: якщо glossary отримає provenance (вище), і якщо SIMARGL проіндексує ще й задачі/коміти — зв'язок терм → історія змін стає **детермінованим пошуком, не RAG-здогадкою**:

1. З provenance факту відомий точний шлях файлу на момент індексації.
2. `git log --follow <file>` або пряма вибірка з `RAWDATA.PATH`/`TASK` таблиць codeXtract (той самий датасет, що живить exp0-exp3) дають точний список задач/комітів, що торкались саме цього файлу.
3. Жодного embedding similarity — це те, що фізично сталося з файлом, а не ймовірнісний здогад.

Формат — окрема секція **в самому кінці статті**, навмисно не для читання підряд, а для пошуку:

```markdown
## Change History
- [SQLFLUFF-1234] Fixed line_position handling for nested WHERE clauses
  commit abc123f, 2024-03-12, +12/-4 layout.py
- [SQLFLUFF-5678] Added `:strict` modifier support
  commit def456a, 2024-06-01, +34/-2 layout.py, rules/L007.py
```

Компактний огляд зверху (БЕС-стиль, швидко читається), історія змін знизу — розростається "як Канада", але це прийнятно саме тому що encyclopedia свідомо не обмежена, і ця секція шукається, а не читається.

**Чому не через glossary напряму**: обробка задач/діффів як `FACTS:` фактично перенесла б увесь Jira в компактний glossary — надто важко для інструменту, чия цінність саме в компактності. Encyclopedia — правильне місце саме тому, що межі розміру немає.

## Відкриті питання і необхідні вдосконалення

1. **Provenance в glossary — перший крок, не опційний.** Без нього обидва підходи до статті і Change History не мають опори. Потребує зміни `_write_term_file`/`_read_term_file` формату (додати source-поле до кожного факту/визначення) — сумісність зі старими файлами без provenance (порожнє/відсутнє поле).
2. **Надійність RAG-grounding для термів/фактів як запитів — не виміряна**, потребує власного експерименту на реальному корпусі перед тим, як довіряти цьому шляху в продакшн-використанні encyclopedia.
3. **Зв'язок задача↔файл потребує самого індексу задач у SIMARGL-форматі** для проєкту, що документується (тут — sqlfluff). Якщо такого датасету немає — Change History секція недоступна, лишається тільки provenance-based цитування джерела документації.
4. **Синонімія все ще нерозв'язана** (успадковано з GLOSSARY.md: "task-to-code retrieval" vs "TCR" — різні терми). Для encyclopedia це болючіше, ніж для glossary — стаття про "TCR" і стаття про "task-to-code retrieval" можуть розвиватись незалежно й розходитись.
5. **Хто вирішує, коли підхід 1 (копіювання) недостатній і потрібен підхід 2 (синтез)?** Найпростіша евристика — по бакету файлу (проза → підхід 1, код → підхід 2), як і в самому glossary, але не перевірено на реальних змішаних корпусах.
