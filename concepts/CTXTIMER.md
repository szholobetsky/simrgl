# CTXTIMER — Вимірювання порогу контексту для локальних моделей

## Мотивація

При роботі з локальними LLM на обмеженому залізі (CPU/GPU з обмеженою пам'яттю та пропускною спроможністю) виникає нетривіальна проблема: модель не "падає" гладко при зростанні контексту, а раптово **перестає встигати** повернути перший токен до таймауту.

Наприклад:
- **gemma-e4b (matryoshka)**: працює швидко до ~8K токенів, на 10K токенів уже не встигає за 360s
- **qwen3.6:35b-a3b (sparse MoE)**: на 10-20 токенах мислить миттєво, на 3000 токенів потрібна істотно довша пауза

Немає універсальної формули — потрібно виміряти емпірично для кожної моделі на кожному залізі.

## Теорія: Чому це відбувається

### Механізм 1: Attention Scaling (всі моделі)

Трансформер обраховує attention scores як квадратичну залежність від довжини контексту:
- Лінійні проекції (Q/K/V, FFN): **O(n · d²)** — лінійно від контексту
- Attention scores: **O(n² · d)** — **квадратично** від контексту

Де n = довжина контексту, d = розмір приховуємого шару (~2500-3000 для 4B моделей).

**Наслідок**: поки n < d, kvadratичний доданок ще малий. Як тільки n наближається до порядку d (8-10K для типової 4B моделі), здійснюється перехід від лінійної до супер-лінійної вартості. Це видно як "коліно" на графіку вартості.

### Механізм 2: KV Cache Memory Bandwidth (особливо на CPU)

Ключова-значень кеш (KV cache) містить збережені активації для кожного попереднього токена:
- На малих контекстах KV cache вміщується в L2/L3кеш процесора (~16-64 МБ)
- Як тільки контекст зростає (навіть на кілька тисяч токенів), KV cache переходить в оперативну пам'ять
- На CPU це стає **memory-bandwidth bottleneck**, не обчислювальним

**Наслідок**: у вас достатньо вільної RAM (32GB дарма), але **пропускна спроможність читання** — це інший ресурс. Кожен новий токен мусить прочитати весь KV cache з RAM, і на CPU це гараздо повільніше, ніж генерація самого токена.

### Механізм 3: Sparse MoE Coverage Saturation (тільки для MoE моделей)

Моделі з динамічною маршрутизацією експертів (як qwen3.6:35b-a3b) вибирають top-k експертів **окремо для кожного токена**:
- На контексті в 10-20 токенів покривається лиш невелика частка всіх експертів → активуються ~3B з 35B параметрів
- На контексті в 3000 токенів різні позиції покривають більшість експертів → реально потрібно прочитати ~35B параметрів, не 3B
- Після точки насичення (коли покривається майже вся модель) подальше зростання контексту вже майже не впливає на час prefill

**Наслідок**: qwen3.6 на 3K контексту "платить" вартість 35B-параметрової моделі, тому й гальмує набагато раніше, ніж gemma (який має фіксовану підмережу 4B).

## Що вимірюємо і чому

### Ollama API: `prompt_eval_duration`

Ollama при генерації повертає точні таймінги:
```json
{
  "prompt_eval_count": 2500,
  "prompt_eval_duration": 45000000000,  // наносекунди
  "eval_count": 1,
  "eval_duration": 200000000
}
```

`prompt_eval_duration` — це час на **prefill**, тобто час до появи першого токена. Саме це нас цікавить.

### Метрика: Максимальне N при таймауті T

Нас цікавить: **яке максимальне N (розмір контексту в токенах), при якому модель встигає почати генерацію до таймауту?**

Це критично для практичного використання:
- Якщо timeout = 360s, і модель потребує 400s на prefill — користувач так ніколи не побачить ні першого токена
- Якщо модель почала в 340s — користувач вже бачить, що "щось працює", і може чекати додаток

## Flow: `/flow ctxtimer`

### Параметри

```
/flow ctxtimer --start 1000 --step 1000 --seq
/flow ctxtimer --start 1000 --end 20000 --bin
```

- `--start N` — з якого розміру контексту починаємо (за замовчуванням: 1000)
- `--end N` — максимум (тільки для `--bin`); контекст, який точно НЕ спрацює
- `--step N` — крок дискретизації при послідовному пошуку (за замовчуванням: 1000)
- `--seq` — послідовна перевірка всіх розмірів (100% вичерпний результат, повільний)
- `--bin` — бінарний пошук (швидко, ~log₂ кількості тестів)

### Як працює

1. **Генерація базової промпти**: створюємо один великий текст англійською мовою (~20K+ токенів) — список фактів, документація, код, текст статті, щось довге і осмислене
2. **Тестування**: для кожного розміру N беремо **перші N токенів** з базової промпти
3. **Виклик LLM**: відправляємо промпт в 1bcoder (який вже знає модель, хост, таймаут)
4. **Вимірювання**: чи встигли ми отримати перший токен до таймауту?
5. **Пошук межі**:
   - **Sequential** (`--seq`): перевіряємо --start, --start+--step, --start+2*--step, ... поки не знайдемо першу невдачу
   - **Binary** (`--bin`): звужуємо інтервал [--start, --end] аж поки не локалізуємо точну межу

## Практичне використання: Map-Reduce архітектура

Дане вимірювання слугує основою для правильної архітектури многошагових агентів на локальному залізі.

### Проблема: "Безпечна" модель ≠ "Розумна" модель

- qwen3.6:35b-a3b — дуже розумна (35B), але только на малих контекстах (<3K)
- gemma-e4b — менш розумна (ефективно 4B), але стійка до великих контекстів (до 8K)
- Локальні dense-моделі 7-8B — компроміс, але тільки якщо контекст < ~5-7K

### Рішення: Двохрівнева архітектура

```
┌─────────────────────────────────────────┐
│  ORCHESTRATOR (Dense, 7-8B)             │
│  Роль: збір контексту, стиснення        │
│  - RAG виклики                          │
│  - Парсинг документації                 │
│  - Виконання тестів                     │
│  - Вихід з інструментів                 │
│  - Резюмування в 1000-2000 токенів      │
└────────────┬────────────────────────────┘
             │ (стиснутий контекст, <2K)
             ↓
┌──────────────────────────────────────────┐
│  REASONER (MoE, 35B)                     │
│  Роль: глибоке міркування                │
│  - Прийме короткий, але детальний input  │
│  - Повернути розв'язок або уточнення    │
└────────────────────────────────────────────┘
```

### Сценарій: Пошук причини регресії в коді

1. **Orchestrator** (dense-модель):
   - Отримує: "Чому `test_user_auth` помер?"
   - Збирає: лог помилки, послідні коміти, файл тесту, пов'язаний код
   - Усього ~8K токенів вхідних даних
   - Резюмує: "Регресія в модулі X, причина — недостатньо валідація Y, лог показує Z"
   - Вихід: 1500 токенів

2. **Reasoner** (MoE-модель):
   - Отримує: 1500-токенне резюме
   - Міркує: як це вирішити, які тест-кейси додати, як це впливає на архітектуру
   - Повертає: детальний план виправлення

**Переваги**:
- Dense модель швидко обробляє будь-яку кількість вхідних даних (немає ефекту насичення експертів)
- MoE модель платить за розумність один раз, на короткому, добре структурованому вході
- Загальний час предбачуваніший, контекст вимірюється ctxtimer

### Нюанси

1. **Точка насичення може бути вищою**: ваш тест може показати, що 35B модель тягне не 3K, а 1.5-2K токенів за прийнятний час
2. **Стиснення — це втрата інформації**: резюме мусить бути якісним, інакше reasoner робітиме над неповною картиною
3. **Ітеративність**: якщо reasoner просить уточнення ("а що в файлі X?"), це знову цикл збирання→стиснення→міркування

## Імплементація

Див. `/flow ctxtimer` в 1bcoder.
