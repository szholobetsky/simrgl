Not swapping — you're right to rule that out (0 swap used, plenty of free RAM). What you're hitting is intrinsic to how transformer attention scales, made worse by being CPU-only.

Why the cliff around 8-10k tokens:                                                                                   
1. Attention cost isn't linear. The linear projections (Q/K/V, FFN) scale O(n · d²). The attention score computation scales O(n² · d). For a ~4B model, hidden dim d is roughly 2500-3000. Once n (context length) approaches that order o magnitude, the n² term stops being negligibltotal compute grows faster than linearlyright around the range you're seeing (8-10k). Below that, you're in the "linear-feeling" zone; above it, each         additional token costs disproportionately mo
2. KV cache stops fitting in cache. At small context the KV cache (and working set) fits in L2/L3 cache. Once it grows past that (a few thousand tokens easily blows a typical few-MB-to-few-tens-of-MB cache), every generated token has to stream the full KV cache from RAM. On CPU this becomes memory-bandwidth bound, not compute bound — you have plenty of free RAM capacity but bandwidth is a different resource than capacity, and it doesn't show up in free/top as pressure. This is a much better fit for "fast then suddenly much slower" than a smooth degradation would be.          3. Decode cost per token also grows with conerate has to attend over all prior KV cacheentries. So even ignoring prefill, tokens/sec drops as the conversation grows — this is expected, not a bug.
                                                                                                                      top showing llama-server pinned at ~397% con ~4 threads, consistent with bothexplanations.                                                                                                        
Practical mitigations, roughly in order of impact:

- Quantize the KV cache: set OLLAMA_KV_CACHE_TYPE=q8_0 (or q4_0 for more aggressive) before starting ollama serve. This shrinks the KV cache 2-4x, pushing the "fits in cache" cliff further out and reducing memory bandwidth per token.
- Enable flash attention: OLLAMA_FLASH_ATTENTION=1. Reduces memory traffic for the attention computation, helps more  as context grows.
- Trim your context — if you're accumulating full conversation history, truncate/summarize older turns instead of resending everything every time.                                                                                      - Check num_thread — make sure it matches phs; oversubscription can hurt throughput onceyou're compute-bound.                                                                                                 - Raise your client timeout or switch to stru get the first token before the fullresponse completes — 360s for a CPU box at large context is genuinely tight for a MoE-ish 4B model.

Формула для грубої оцінки

T_prefill(n) ≈ max(
    compute_bound:  2 · active_params · n /
    memory_bound:   bytes_touched(n) / RAM_bandwidth
)

де bytes_touched(n) росте від active_params_bytes (малий n) до total_params_bytes (n після точки насичення покриття
експертів), плюс окремий доданок O(n²·d) на  моделей.

Порахувати теоретично складно (точка насичення залежить від кількості експертів, top-k, розподілу маршрутизації), але виміряти емпірично — легко, і це набагато надійніше за формулу.
